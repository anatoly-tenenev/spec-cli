CLI
  spec-cli is a schema-aware CLI for working with specification entities as JSON-like documents.

Schema
  Workspace: /spec-cli-fixed-root/workspace
  ResolvedPath: /spec-cli-fixed-root/spec.schema.yaml
  Status: loaded

Execution model
  `schema check` validates the schema language and reports compile diagnostics only.
  `validate` validates workspace entities against the effective schema in one deterministic run.
  `query`, `get`, `add`, `update`, and `delete` use schema-derived command models.
  `help` remains tolerant; when schema-derived values are unavailable, do not infer them heuristically.

Specification model
  A specification is a collection of typed entities.
  Each entity is a logical JSON-like document with built-ins, meta, refs, and content.sections.
  Storage details such as filesystem paths, markdown layout, frontmatter, and headings are internal details and are not part of the primary user model.

Projection conventions
  The specification projection uses JSON Schema-like structure plus spec-cli extensions.
  Standard JSON Schema keywords keep their usual meaning.
  spec-cli extensions:
    - x-presence: required | optional | conditional
    - x-requiredWhen: condition for conditional presence, evaluated over the logical entity document model
    - x-kind: logical value kind not expressed by base JSON Schema
    - x-refTypes: allowed target entity types for entityRef values
    - x-const: dynamic const as {"kind":"interpolation","source":"<original-source>"}
    - x-enum: dynamic or mixed enum as ordered [{"kind":"literal","value":...}|{"kind":"interpolation","source":"<original-source>"}]
  Native const/enum are emitted only for fully static literal constraints.
  content.sections.<section_name>.title is canonical scalar heading title; only the first schema title is published.
  x-kind=entityRef describes the schema-level meaning of a reference field.
  Read-side ref objects exposed by query/get are described separately in Reference value model.

Reference value model
  Schema-level meaning:
    - entityRef is a logical link to another entity
    - scalar/array ref cardinality is encoded by projection shape (x-kind on scalar refs; type=array + items.x-kind on array refs)
    - x-refTypes and x-presence define allowed target types and presence
  Read-side projection:
    - query/get expose scalar refs as {id, type, slug, resolved, reason}
    - query/get expose array refs as arrays of the same ref objects
  Write-side form:
    - add/update accept target ids for refs.<ref_field>
    - array ref fields accept arrays of target ids

Global options
  --workspace <path>: optional; default "."; root workspace path
  --schema <path>: optional; default "spec.schema.yaml"; effective schema path
  --config <path>: optional; JSON config with "schema"/"workspace"; if omitted, auto-loads "./spec-cli.json" when present
  --format <json|text>: optional; default "json"; help supports only text output; explicit --format json for help returns CAPABILITY_UNSUPPORTED
  --require-absolute-paths: optional; reject explicit relative filesystem paths

Commands
  help: show deterministic text help for the CLI and its document model
  schema: inspect schema compile diagnostics
  validate: validate schema and workspace entities in one deterministic run
  query: read many entities through a schema-derived read model
  get: read one entity through a schema-derived read model
  add: create one entity through a schema-derived write model
  update: patch one entity through a schema-derived write model
  delete: delete one entity subject to reference integrity
  version: show CLI version

Command details

  Operation model

    Renders one schema-derived overview for the whole invocation.

      repeatable: false

      required: false
      repeatable: false

    - help --format json returns CAPABILITY_UNSUPPORTED.
  Examples (syntax-only)

query
  Syntax
    spec-cli query [options]
  Operation model
    query reads many entities through one schema-derived query-item read model.
    --type restricts the active entity-type set before planning and filtering.
    --select projects fields from that read model.
    --sort orders results by sortable read-model paths.
    --where evaluates JMESPath truthiness over the same read model.
  Active type set
    If `--type` is omitted, all entity types from the specification projection are active.
    If `--type` is repeated, only the selected entity types are active.
    The query-item read model is the union over the active type set.
    Path-based select, sort, and scalar ref detail paths must be compatible across the active type set.
  Read model
    built-ins:
      - type
      - id
      - slug
      - revision
      - createdDate
      - updatedDate
    type-local fields:
      - meta.<meta_field> from the active type set
      - refs.<ref_field> from the active type set
      - content.raw from the active type set
      - content.sections.<section_name> from the active type set
    The exact type-local field catalog is defined by the Specification projection.
  Read-model path forms
    built-in paths:
      - type
      - id
      - slug
      - revision
      - createdDate
      - updatedDate
    meta paths:
      - meta.<meta_field>
    scalar ref paths:
      - refs.<scalar_ref_field>
      - refs.<scalar_ref_field>.id|type|slug|resolved|reason
    array ref paths:
      - refs.<array_ref_field>
      - query expressions may inspect array items as ref objects with id|type|slug|resolved|reason
    content paths:
      - content.raw
      - content.sections
      - content.sections.<section_name>
  Where language
    `--where` accepts a JMESPath expression.
    spec-cli does not add project-specific operators or helper functions on top of JMESPath.
    Filtering uses JMESPath truthiness; the expression result does not need to be boolean.
    `createdDate` and `updatedDate` use date comparison semantics in the where model.
    literal forms (syntax only):
      - string: '<string_value>'
      - number: `42`
      - null: `null`
  Operation constraints
    select:
      - built-in paths
      - meta paths
      - refs
      - refs.<ref_field>
      - scalar ref detail paths refs.<scalar_ref_field>.id|type|slug|resolved|reason
      - content.raw
      - content.sections
      - content.sections.<section_name>
    sort:
      - built-in scalar paths
      - meta.<meta_field>
      - scalar ref detail paths refs.<scalar_ref_field>.id|type|slug|resolved|reason
      - content.raw
      - content.sections.<section_name>
    where:
      allowed paths:
        - built-in paths
        - meta.<meta_field>
        - refs.<ref_field>
        - scalar ref detail paths refs.<scalar_ref_field>.id|type|slug|resolved|reason
        - content.sections.<section_name>
      forbidden:
        - content root except content.sections...
        - content.raw
        - meta.<ref_field> access for entityRef values
      runtime notes:
        - evaluates against the same query-item read model
        - optional and conditional fields may be absent at runtime
        - use guards or fallbacks when needed
  Defaults
    default projection:
      - type
      - id
      - slug
      - meta
      - refs
    default order:
      - type:asc
      - id:asc
  Options
    --type <entity_type>
      takes_value: true
      required: false
      repeatable: true
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: entity type keys from the specification projection
      description: Restrict the active entity-type set.
    --where <expr>
      takes_value: true
      required: false
      repeatable: false
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: JMESPath expression over the query-item read model
      description: Evaluate a JMESPath filter over the query-item read model.
    --select <path>
      takes_value: true
      required: false
      repeatable: true
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: selectable paths from the query-item read model
      description: Select a path from the query-item read model.
    --sort <path[:asc|desc]>
      takes_value: true
      required: false
      repeatable: true
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: sortable paths from the query-item read model
      description: Sort by a sortable query-item read-model path.
    --limit <n>
      takes_value: true
      required: false
      repeatable: false
      mutually_exclusive_groups: none
      schema_derived: false
      schema_derivation: none
      description: Page size; integer >= 0; default 100.
    --offset <n>
      takes_value: true
      required: false
      repeatable: false
      mutually_exclusive_groups: none
      schema_derived: false
      schema_derivation: none
      description: Page offset; integer >= 0; default 0.
  Rules
    - If --type is omitted, all entity types from the specification projection participate.
    - Only paths compatible with the active type set may be selected, sorted, or used in --where.
    - If --sort is omitted, effective sort is type:asc, then id:asc.
    - If --sort is provided, effective sort is user sort plus hidden tail type:asc, then id:asc, unless that exact tail is already present at the end.
    - entityRef values are addressed through refs.<ref_field>, not meta.<ref_field>.
    - Selected leaf paths under meta.<meta_field>, refs.<ref_field>, and content.sections.<section_name> materialize as null when absent.
    - Aggregate selectors meta, refs, and content.sections keep sparse objects and do not synthesize unknown children.
  Examples (syntax-only)
    spec-cli query --type <entity_type> --select id --select meta.<meta_field>
    spec-cli query --select refs.<ref_field> --select content.sections.<section_name>
    spec-cli query --where "meta.<meta_scalar_field> == '<string_value>'"
    spec-cli query --where "refs.<scalar_ref_field>.id == '<entity_id>'"
    spec-cli query --where "contains(content.sections.<section_name> || '', '<substring>')"
    spec-cli query --where "length(refs.<array_ref_field>[?reason == '<reason_value>']) > `0`"
    spec-cli query --sort updatedDate:desc --limit 50 --offset 0
    Available paths, value kinds, and ref cardinality depend on the effective schema.

get
  Syntax
    spec-cli get [options] --id <entity_id>
  Operation model
    get reads one entity through a schema-derived single-entity read model.
    --id selects the target entity.
    --select projects fields from the single-entity read model.
  Target type
    The single-entity read model is determined by the resolved target entity type.
    Built-ins are always present.
    Type-local meta, refs, content.raw, and content.sections fields come from the target type in the Specification projection.
  Read model
    built-ins:
      - type
      - id
      - slug
      - revision
      - createdDate
      - updatedDate
    type-local fields:
      - meta.<meta_field> from the target entity type
      - refs.<ref_field> from the target entity type
      - content.raw from the target entity type
      - content.sections.<section_name> from the target entity type
    entityRef values are exposed through refs.<ref_field> and are not projected under meta.<ref_field>.
  Read-model path forms
    built-in paths:
      - type
      - id
      - slug
      - revision
      - createdDate
      - updatedDate
    meta paths:
      - meta.<meta_field>
    scalar ref paths:
      - refs.<scalar_ref_field>
      - refs.<scalar_ref_field>.id|type|slug|resolved|reason
    content paths:
      - content.raw
      - content.sections
      - content.sections.<section_name>
  Operation constraints
    select:
      - built-in paths
      - meta paths
      - refs
      - refs.<ref_field>
      - scalar ref detail paths refs.<scalar_ref_field>.id|type|slug|resolved|reason
      - content.raw
      - content.sections
      - content.sections.<section_name>
  Defaults
    if --select is omitted, default projection:
      - type
      - id
      - slug
      - meta
      - refs
  Options
    --id <entity_id>
      takes_value: true
      required: true
      repeatable: false
      mutually_exclusive_groups: none
      schema_derived: false
      schema_derivation: none
      description: Exact target id.
    --select <path>
      takes_value: true
      required: false
      repeatable: true
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: selectable paths from the single-entity read model
      description: Select a path from the single-entity read model.
  Rules
    - Selected leaf paths under meta.<meta_field>, refs.<ref_field>, and content.sections.<section_name> materialize as null when absent.
    - Aggregate selectors meta, refs, and content.sections keep sparse objects and do not synthesize unknown children.
    - Existing but invalid target data must not degrade into false ENTITY_NOT_FOUND when deterministic read is still possible.
  Examples (syntax-only)
    spec-cli get --id <entity_id>
    spec-cli get --id <entity_id> --select id --select meta.<meta_field>

add
  Syntax
    spec-cli add --type <entity_type> --slug <slug> [options]
  Operation model
    add creates one logical entity document through a schema-derived write model.
    Path-based writes target writable logical fields.
    Whole-body content input is a separate input mode, not the primary document model.
  Identity and system fields
    `type` is chosen by `--type`.
    `slug` is chosen by `--slug`.
    `id` is derived by the command from the new entity type, the provided slug, and the effective schema.
    `createdDate`, `updatedDate`, and `revision` are command-managed built-ins and are not set directly.
  Write model
    writable paths:
      - meta.<meta_field>
      - refs.<ref_field>
      - content.sections.<section_name>
    whole-body operations:
      - --content-file
      - --content-stdin
  Value syntax
    `--set` uses `<path=value>`.
    Shell quoting applies outside the argument; for non-ref metadata fields the right-hand side is parsed as a YAML value.
    Value forms (syntax only)
      - string scalar meta: `meta.<meta_scalar_field>=<string_value>`
      - boolean scalar meta: `meta.<meta_scalar_field>=true`
      - number scalar meta: `meta.<meta_scalar_field>=42`
      - array meta: `meta.<meta_array_field>='[<value_1>, <value_2>]'`
      - scalar entityRef: `refs.<scalar_ref_field>=<entity_id>`
      - array entityRef: `refs.<array_ref_field>='[<entity_id_1>, <entity_id_2>]'`
    Actual writable paths, value kinds, and ref cardinality depend on the effective schema.
    Empty value after `=` is treated as an empty string and may still fail schema type checks.
    `content.sections.<section_name>` values are treated as raw section text, not YAML-decoded values.
  Whole-body heading contract
    Whole-body serializer uses canonical headings in the form `## <title> {#<section_name>}`.
    In the Specification projection, `content.sections.<section_name>.title` is the canonical heading title used by default.
    If the schema defines multiple titles for one section, help publishes only the first canonical title.
  Options
    --type <entity_type>
      takes_value: true
      required: true
      repeatable: false
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: entity type keys from the specification projection
      description: Target entity type.
    --slug <slug>
      takes_value: true
      required: true
      repeatable: false
      mutually_exclusive_groups: none
      schema_derived: false
      schema_derivation: none
      description: Entity slug used to derive the new entity identity in the effective schema.
    --set <path=value>
      takes_value: true
      required: false
      repeatable: true
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: writable paths from the entity write model
      description: Set a writable logical field by path.
    --set-file <path=file_path>
      takes_value: true
      required: false
      repeatable: true
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: section paths from the entity write model
      description: Set a content section body from file content.
    --content-file <path>
      takes_value: true
      required: false
      repeatable: false
      mutually_exclusive_groups: content_body_source
      schema_derived: false
      schema_derivation: none
      description: Whole-body content source from filesystem path.
    --content-stdin
      takes_value: false
      required: false
      repeatable: false
      mutually_exclusive_groups: content_body_source
      schema_derived: false
      schema_derivation: none
      description: Whole-body content source from stdin.
    --dry-run
      takes_value: false
      required: false
      repeatable: false
      mutually_exclusive_groups: none
      schema_derived: false
      schema_derivation: none
      description: Run validation and planning without filesystem mutation.
  Rules
    - Allowed write paths are derived from the effective schema.
    - Reference writes use target entity ids, not read-side ref objects produced by query/get.
    - --set-file is allowed only for content.sections.<section_name>.
    - Whole-body inputs are not write-model paths.
    - Whole-body serializer emits canonical headings as `## <title> {#<section_name>}`.
    - Specification projection `title` fields expose canonical heading titles used by default.
    - Whole-body inputs may be combined with meta.<meta_field> and refs.<ref_field> writes.
    - Whole-body inputs cannot be combined with content.sections.<section_name> path-based operations.
    - Built-in fields and aggregate content paths are not writable: type, id, slug, createdDate, updatedDate, content, content.raw, content.sections.
    - --content-file and --content-stdin are mutually exclusive.
  Examples (syntax-only)
    spec-cli add --type <entity_type> --slug <slug> --set meta.<meta_scalar_field>=<string_value>
    spec-cli add --type <entity_type> --slug <slug> --set refs.<scalar_ref_field>=<entity_id>
    spec-cli add --type <entity_type> --slug <slug> --set refs.<array_ref_field>='[<entity_id_1>, <entity_id_2>]'
    spec-cli add --type <entity_type> --slug <slug> --content-file ./input/body.md --set meta.<meta_scalar_field>=<string_value> --set refs.<scalar_ref_field>=<entity_id>
    spec-cli add --type <entity_type> --slug <slug> --set-file content.sections.<section_name>=./input/section.md
    spec-cli add --type <entity_type> --slug <slug> --content-file ./input/body.md

    - Reference writes use target entity ids, not read-side ref objects produced by query/get.

delete
  Syntax
    spec-cli delete [options] --id <entity_id>
  Operation model
    delete removes one entity by exact id, subject to inbound reference integrity.
  Reference integrity model

    both scalar and array entityRef links participate in blocking checks.
