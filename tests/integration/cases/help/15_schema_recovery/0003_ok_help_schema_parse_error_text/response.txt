CLI
  spec-cli is a machine-first CLI utility for spec documents.

Global options
  --workspace <path>: optional; default "."; root workspace filesystem path
  --schema <path>: optional; default "spec.schema.yaml"; effective schema filesystem path
  --config <path>: optional; JSON config with "schema"/"workspace"; if omitted, auto-loads "./spec-cli.json" when present
  --format <json|text>: optional; default "json"; help supports only text output; explicit --format json for help returns CAPABILITY_UNSUPPORTED
  --require-absolute-paths: optional; reject explicit relative filesystem paths

Commands
  help: show deterministic text help for CLI and commands
  query: read many entities with structural filters and deterministic pagination
  get: read one entity by exact id
  add: create a new entity
  update: patch one entity by exact id
  delete: delete one entity by exact id
  validate: validate schema and workspace documents
  version: show CLI version

Schema
  ResolvedPath: /spec-cli-fixed-root/spec.schema.yaml
  Status: invalid
  ReasonCode: SCHEMA_PARSE_ERROR
  Impact: schema-derived entity types, namespace paths, enum values and CLI projection of schema-derived fields are unavailable; do not infer these values heuristically
  RecoveryClass: fix_schema_file
  RetryCommand: spec-cli validate --schema /spec-cli-fixed-root/spec.schema.yaml

Command details
help
  Syntax
    spec-cli help
    spec-cli help <command>
  Options
    <command>
      required: false
      repeatable: false
      description: Optional command name for detailed help.
  Rules
    - Returns stable text-first output.
    - Renders exactly one Schema section per invocation.
    - Schema always includes ResolvedPath and Status; when schema is unavailable it includes ReasonCode, Impact, RecoveryClass and RetryCommand.
    - help --format json returns CAPABILITY_UNSUPPORTED.
  Examples
    spec-cli help
    spec-cli help query

query
  Syntax
    spec-cli query [options]
  Options
    --type <entity_type>
      takes_value: true
      required: false
      repeatable: true
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: entity type keys from Schema.entity
      description: Early entity type filter.
    --where-json <json>
      takes_value: true
      required: false
      repeatable: false
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: where-json.field paths from CLI filter-namespace
      description: Structural filter JSON expression.
    --select <field>
      takes_value: true
      required: false
      repeatable: true
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: selectors from CLI projection-namespace
      description: Projection-namespace selector.
    --sort <field[:asc|desc]>
      takes_value: true
      required: false
      repeatable: true
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: sort fields from CLI filter-namespace
      description: Deterministic sort term.
    --limit <n>
      takes_value: true
      required: false
      repeatable: false
      mutually_exclusive_groups: none
      schema_derived: false
      schema_derivation: none
      description: Page size; integer >= 0; default 100.
    --offset <n>
      takes_value: true
      required: false
      repeatable: false
      mutually_exclusive_groups: none
      schema_derived: false
      schema_derivation: none
      description: Page offset; integer >= 0; default 0.
  Rules
    - projection-namespace argument is --select.
    - filter-namespace arguments are --sort and where-json.field.
    - projection-namespace selectors: type, id, slug, revision, created_date, updated_date, meta.<name>, refs, refs.<field>, content.raw, content.sections, content.sections.<name>.
    - filter-namespace leaf paths: type, id, slug, revision, created_date, updated_date, meta.<name>, refs.<field>.resolved, refs.<field>.type, refs.<field>.id, refs.<field>.slug, content.raw, content.sections.<name>.
    - --type uses schema-derived entity type keys and is not part of projection-namespace or filter-namespace.
    - If --select is omitted, default projection is type, id, slug, meta and refs.
    - Query guarantees deterministic ordering.
    - If --sort is omitted, effective sort is type:asc, then id:asc.
    - If --sort is provided, effective sort is user sort plus hidden tail type:asc, then id:asc, unless that tail is already present at the end.
    - where-json logical nodes: {"op":"and","filters":[...]}, {"op":"or","filters":[...]}, {"op":"not","filter":{...}}.
    - where-json leaf node: {"field":"meta.status","op":"eq","value":"active"}.
    - Leaf operators: eq, neq, in, not_in, exists, not_exists, gt, gte, lt, lte, contains.
    - exists/not_exists do not use value; in/not_in require array value; all other leaf operators require value.
    - Allowed where-json fields come from filter-namespace.
    - content.raw is forbidden in where-json even though it is available in projection-namespace and --sort.
    - For content.sections.<name> where-json supports contains, exists and not_exists only.
    - gt/gte/lt/lte are valid only for numbers and YYYY-MM-DD dates; dates compare lexicographically.
    - Comparison with incompatible type returns INVALID_QUERY, not false.
    - For enum fields, eq/neq/in/not_in values must belong to enum with case sensitivity, otherwise INVALID_QUERY.
    - Missing value semantics: exists returns true only when value is present; not_exists returns true only when value is absent; all other leaf operators return false for missing value.
  Examples
    spec-cli query --type feature --select id --select meta.status
    spec-cli query --where-json '{"field":"meta.status","op":"eq","value":"active"}'
    spec-cli query --where-json '{"op":"and","filters":[{"field":"meta.status","op":"eq","value":"active"},{"field":"refs.owner.id","op":"eq","value":"SVC-2"}]}'
    spec-cli query --where-json '{"field":"content.sections.summary","op":"contains","value":"retry"}'
    spec-cli query --where-json '{"field":"refs.owner.id","op":"eq","value":"SVC-2"}'
    spec-cli query --sort updated_date:desc --limit 50 --offset 0

get
  Syntax
    spec-cli get [options] --id <entity_id>
  Options
    --id <entity_id>
      takes_value: true
      required: true
      repeatable: false
      mutually_exclusive_groups: none
      schema_derived: false
      schema_derivation: none
      description: Exact target id.
    --select <field>
      takes_value: true
      required: false
      repeatable: true
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: selectors from CLI projection-namespace
      description: Projection-namespace selector.
  Rules
    - Projection-namespace selectors: type, id, slug, revision, created_date, updated_date, meta.<name>, refs, refs.<field>, content.raw, content.sections, content.sections.<name>.
    - --select uses projection-namespace selectors.
    - --id addresses one entity by exact id and returns a single-entity read result.
    - If --select is omitted, default projection is type, id, slug, meta and refs.
    - entity_ref scalar fields are exposed through refs.<field> and are not projected under meta.<name>.
    - If selected content.sections.<name> is absent, the field must be present with null value.
    - Existing but invalid target data must not degrade into false ENTITY_NOT_FOUND when deterministic read is still possible.
  Examples
    spec-cli get --id FEAT-8
    spec-cli get --id FEAT-8 --select id --select meta.status

add
  Syntax
    spec-cli add --type <entity_type> --slug <slug> [options]
  Options
    --type <entity_type>
      takes_value: true
      required: true
      repeatable: false
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: entity type keys from Schema.entity
      description: Target entity type.
    --slug <slug>
      takes_value: true
      required: true
      repeatable: false
      mutually_exclusive_groups: none
      schema_derived: false
      schema_derivation: none
      description: Entity slug for id/path derivation.
    --set <path=value>
      takes_value: true
      required: false
      repeatable: true
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: write paths from CLI write-namespace
      description: Set writable slot value by write-namespace path.
    --set-file <path=file_path>
      takes_value: true
      required: false
      repeatable: true
      mutually_exclusive_groups: none
      schema_derived: true
      schema_derivation: content.sections.<name> from raw schema sections
      description: Set section body from file content.
    --content-file <path>
      takes_value: true
      required: false
      repeatable: false
      mutually_exclusive_groups: content_body_source
      schema_derived: false
      schema_derivation: none
      description: Whole-body content source from filesystem path.
    --content-stdin
      takes_value: false
      required: false
      repeatable: false
      mutually_exclusive_groups: content_body_source
      schema_derived: false
      schema_derivation: none
      description: Whole-body content source from stdin.
    --dry-run
      takes_value: false
      required: false
      repeatable: false
      mutually_exclusive_groups: none
      schema_derived: false
      schema_derivation: none
      description: Run validation and planning without filesystem mutation.
  Rules
    - Write-namespace arguments are --set and --set-file.
    - Write-namespace paths: meta.<name>, refs.<field>, content.sections.<name>.
    - Projection from raw schema: non-entity_ref metadata -> meta.<name>; entity_ref metadata (scalar or array items.type=entity_ref) -> refs.<field>; sections -> content.sections.<name>.
    - Allowed write-paths are derived from the effective schema.
    - refs.<field> values are target entity ids (scalar id or YAML array of ids).
    - --set-file is allowed only for content.sections.<name>.
    - Whole-body inputs --content-file and --content-stdin are not write-namespace paths.
    - For whole-body --content-file/--content-stdin input, use headings like `## <title> {#<section_name>}`.
    - Built-in fields and aggregate content paths are not writable: type, id, slug, created_date, updated_date, content, content.raw, content.sections.
    - --content-file and --content-stdin are mutually exclusive.
  Examples
    spec-cli add --type feature --slug login-flow --set meta.status=active
    spec-cli add --type feature --slug login-flow --set refs.owner=SVC-2 --set content.sections.summary='Short summary'
    spec-cli add --type feature --slug login-flow --set meta.tags='[api, reliability]' --set refs.watchers='[SVC-1, SVC-2]'
    spec-cli add --type feature --slug login-flow --set-file content.sections.summary=./input/summary.md
    spec-cli add --type feature --slug login-flow --content-file ./input/body.md
