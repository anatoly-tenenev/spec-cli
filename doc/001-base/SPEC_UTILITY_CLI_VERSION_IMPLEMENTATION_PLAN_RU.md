# План реализации команды `version`

## 1. Цель

Реализовать baseline-команду:

```bash
spec-cli version
```

Ожидаемый нормативный JSON-ответ:

```json
{
  "result_state": "valid",
  "version": "0.1.0"
}
```

Команда должна соответствовать общим правилам CLI по формату вывода: формат задаётся глобальной опцией `--format`, а не command-specific опциями `version`.

## 2. Требования из baseline, влияющие на реализацию

### 2.1. Поведение команды

- `version` возвращает версию утилиты.
- Нормативный контракт команды — JSON-ответ.
- `result_state` обязателен во всех JSON-ответах верхнего уровня.
- Для успешного ответа используется `result_state: "valid"`.
- У `version` нет command-specific опций; любые опции после `version` считаются `INVALID_ARGS`.

### 2.2. Глобальные правила

- CLI вызывается как `spec-cli [global options] <command> [command options]`.
- Формат вывода задаётся глобально через `--format`.
- Поддерживаемое значение глобального формата для baseline: `json`.
- В JSON-режиме данные выводятся в `stdout`, диагностические сообщения — в `stderr`.
- CLI неинтерактивен по умолчанию.

### 2.3. Ошибки и коды завершения

- При невозможности вернуть основной результат в JSON команда должна использовать top-level `error`.
- Минимальные поля ошибки: `code`, `message`, `exit_code`.
- Код завершения `0` используется для успеха.
- Код завершения `2` используется для неверных аргументов CLI или невалидного запроса.
- Код завершения `5` используется для внутренних ошибок утилиты.

## 3. Решения по реализации

### 3.1. Источник версии

Нужно выбрать единый источник строки версии, который:

- доступен без чтения workspace и schema;
- не зависит от доменных данных;
- может использоваться и в коде, и в сборке, и в тестах.

Практичное решение для baseline:

- завести один canonical source version string внутри приложения;
- команда `version` только читает это значение и сериализует его в ответ;
- не вычислять версию динамически из workspace, schema или состояния файлов.

Жесткая policy для baseline:

- источник версии в runtime только один: build-time variable/constant `Version`;
- release-значение `Version` должно приходить из CI/CD или release pipeline;
- рекомендуемый upstream source для release-значения: git tag релизной сборки;
- локальная или нерелизная сборка должна использовать явный fallback, например `dev` или `0.1.0-dev`;
- runtime-код команды `version` не должен вызывать `git`, читать теги, commit hash, файлы workspace или schema для вычисления версии;
- если build pipeline не подставил release-версию, команда возвращает fallback-значение как обычную строку версии сборки.

### 3.2. Контракт команды

Минимальный контракт команды:

- вход: команда `version` и глобальные CLI-опции;
- успешный выход в JSON:
  - `result_state = "valid"`
  - `version = <string>`
- формат управляется только глобальной опцией `--format` по общим правилам CLI.

### 3.3. Границы ответственности

Команда `version` не должна:

- читать workspace;
- загружать schema;
- выполнять валидацию документов;
- зависеть от entity path или read/write namespace;
- возвращать дополнительные поля без явной необходимости baseline.

## 4. Шаги реализации

### Шаг 1. Завести модель версии

- определить место хранения canonical version string;
- реализовать build-time injection для `Version` из CI/release pipeline;
- определить и задокументировать fallback-значение для локальных сборок;
- обеспечить доступ к этой строке из command handler;
- исключить дублирование значения версии в нескольких участках runtime-кода.

Результат шага:

- в кодовой базе есть единый источник версии, возвращающий release-строку вроде `0.1.0` или fallback вроде `dev`.

### Шаг 2. Добавить команду `version` в CLI routing

- зарегистрировать подкоманду `version` в основном CLI parser/router;
- подключить описание команды в help/discovery-слое, если он формируется из регистра команд;
- убедиться, что команда не требует command-specific опций.

Результат шага:

- вызов `spec-cli version` доходит до отдельного обработчика.

### Шаг 3. Реализовать handler команды

- в handler получить canonical version string;
- сформировать success payload:
  - `result_state: "valid"`
  - `version: "<semantic version>"`
- не вводить command-specific управление форматом для `version`.

Результат шага:

- команда стабильно формирует JSON-ответ без обращения к workspace и schema.

### Шаг 4. Встроить общую обработку ошибок

- если parser команды получает неподдерживаемые аргументы, вернуть CLI-ошибку с exit code `2`;
- если global parser получает невалидный глобальный формат, вернуть CLI-ошибку с exit code `2`;
- если сериализация ответа или доступ к internal version provider неожиданно падает, вернуть internal error с exit code `5`;
- в JSON-режиме использовать top-level `error`, только если основной payload вернуть невозможно.

Результат шага:

- поведение согласовано с общим baseline error contract.

### Шаг 5. Добавить тесты

- покрыть нормативный JSON-сценарий;
- покрыть использование глобального `--format json`;
- покрыть ошибки неверных аргументов команды;
- покрыть инвариант отсутствия чтения schema/workspace, если это проверяемо архитектурно или через unit/integration seam.

Результат шага:

- команда защищена от регрессий контракта.

## 5. Тестовый план

### 5.1. Позитивные сценарии

1. `spec-cli version`
   Ожидание:
   - exit code `0`
   - `stdout` содержит валидный JSON
   - JSON содержит только необходимые для baseline поля:
     - `result_state = "valid"`
     - `version` как непустую строку

2. `spec-cli --format json version`
   Ожидание:
   - exit code `0`
   - результат эквивалентен `spec-cli version`

### 5.2. Негативные сценарии

1. `spec-cli version --unknown-flag`
   Ожидание:
   - exit code `2`
   - корректное диагностическое сообщение
   - top-level `error` в JSON

2. `spec-cli version --format json`
   Ожидание:
   - exit code `2`
   - ошибка `INVALID_ARGS`, так как `--format` не является command-specific опцией `version`

3. `spec-cli --format yaml version`
   Ожидание:
   - exit code `2`
   - ошибка неверного значения глобального `--format`

4. Сбой внутреннего version provider
   Ожидание:
   - exit code `5`
   - top-level `error` с обязательными полями

5. Локальная сборка без CI-injection
   Ожидание:
   - exit code `0`
   - команда возвращает заранее определенный fallback, например `dev` или `0.1.0-dev`
   - поведение детерминировано и не зависит от наличия git metadata в runtime

### 5.3. Контрактные проверки

- `result_state` присутствует всегда в успешном JSON-ответе;
- `version` присутствует только как строковое поле;
- команда не добавляет доменных полей, не относящихся к baseline `version`;
- `stderr` не загрязняет `stdout` в JSON-режиме;
- runtime команды не читает git metadata для вычисления версии.

## 6. Критерии готовности

Команда считается реализованной, если:

- `spec-cli version` возвращает нормативный JSON из baseline;
- команда завершает успех с exit code `0`;
- неверные аргументы завершаются с exit code `2`;
- внутренние сбои завершаются с exit code `5`;
- реализация не зависит от workspace и schema;
- добавлены автоматические тесты на JSON и error-path сценарии.

## 7. Порядок внедрения

Рекомендуемая последовательность:

1. Выделить canonical source версии.
2. Настроить build-time injection версии из CI/tag и fallback для локальных сборок.
3. Зарегистрировать команду `version` в CLI parser/router.
4. Реализовать JSON payload команды через общий output pipeline.
5. Подключить обработку ошибок по общему контракту.
6. Добавить unit/integration тесты.
7. Проверить фактический CLI-вывод вручную или автотестом на process level.

## 8. Что не входит в объём этой задачи

- расширение контракта `version` дополнительными метаданными сборки;
- чтение версии из schema или workspace;
- поддержка дополнительных machine-first режимов, кроме baseline JSON;
- добавление отдельного command-specific format-поведения для `version`;
- любые изменения в командах `help`, `validate`, `query`, `get`, `add`, `update`, `delete`.
