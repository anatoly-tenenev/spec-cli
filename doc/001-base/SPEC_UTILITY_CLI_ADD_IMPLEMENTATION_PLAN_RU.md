# Подробный план реализации команды `add`

## 1. Назначение документа

Этот план описывает реализацию команды `spec-cli add` для baseline CLI API и набор интеграционных тестов.
Документ опирается только на:

- `SPEC_UTILITY_CLI_API_BASELINE_RU.md`
- `SPEC_STANDARD_RU_REVISED_V3.md`

Цель плана: довести `add` до предсказуемой machine-first реализации с полным pre-write validation, атомарной записью и черноящичным интеграционным покрытием.

## 2. Что именно должно быть реализовано

Ниже перечислен обязательный контракт, который код должен закрыть.

- Команда вызывается как `spec-cli add [options]`.
- Обязательные опции: `--type <entity_type>` и `--slug <slug>`.
- Поддерживаются опции `--set`, `--set-file`, `--content-file`, `--content-stdin`, `--dry-run`.
- `--content-file` и `--content-stdin` взаимоисключающие.
- `id`, `created_date`, `updated_date` вычисляются автоматически.
- `add` не использует `schema.model` как источник write-контракта.
- Допустимые write-path для `--set` и `--set-file` выводятся напрямую из raw schema:
  - `meta.<field>` для полей из `meta.fields`;
  - `refs.<field>` для полей из `meta.fields`, где `schema.type = entity_ref`;
  - `content.sections.<name>` для секций из `content.sections`.
- `--set-file` разрешен только для `content.sections.<name>`.
- Отдельный контракт `required_set_paths` не используется; обязательность данных определяется полной валидацией итоговой сущности.
- Нормативно запрещенные формы записи определяются baseline-правилами write-namespace и raw schema, без обращения к `schema.model`.
- Один и тот же write-path нельзя передать более одного раза суммарно через `--set` и `--set-file`.
- Перед записью должна выполняться валидация создаваемой сущности целиком.
- При validation-fail команда должна вернуть top-level `error`, а не успешный payload.
- Если доступны структурированные diagnostics, они должны возвращаться в `error.details.validation.issues[]`.
- Запись должна быть наблюдаемо атомарной: при неуспехе состояние на диске не меняется.
- В JSON-ответах верхнего уровня обязательно поле `result_state`.
- Для сущности в ответе обязательно поле `revision`, вычисленное по фактическому состоянию документа.
- Реализация должна уважать правила стандарта для `id_prefix`, `path_pattern`, `required` / `required_when`, `entity_ref`, `content.sections`, `slug`, `created_date`, `updated_date`, closed-world схемы и диагностических классов.

## 3. Решения, которые нужно зафиксировать до кодирования

Baseline и стандарт задают контракт `add`, но оставляют несколько реализационных зазоров. Их лучше зафиксировать в самом начале, иначе поведение получится недетерминированным.

### 3.1. Генерация нового `id`

Стандарт требует только формат `"{id_prefix}-N"` и уникальность `N` в рамках типа. Алгоритм выбора нового `N` не нормирован.

Зафиксированное решение:

- использовать детерминированное правило `max(existing N for type) + 1`;
- не пытаться переиспользовать дырки в последовательности;
- строить индекс существующих `id` один раз на старте команды;
- в baseline не вводить filesystem/workspace lock для `add`.

Причина: правило `max + 1` проще, стабильнее и легче тестируется, чем стратегия "минимальный свободный номер".

### 3.2. Источник даты для `created_date` и `updated_date`

Стандарт требует формат `YYYY-MM-DD`, но не фиксирует источник "сегодняшней" даты.

Зафиксированное решение:

- ввести внедряемый `Clock`;
- в production брать дату как UTC day из одного явного источника времени процесса;
- в тестах подменять `Clock`, чтобы дата была детерминированной;
- одно и то же значение использовать одновременно для `created_date` и `updated_date` при создании.

### 3.3. Парсинг значения в `--set <path=value>`

Baseline указывает `value_kind: "typed_scalar"` / `"target_entity_id"` / `"string"`, но не нормирует, как строка CLI превращается в типизированное значение.

Зафиксированное решение:

- для `value_kind: "target_entity_id"` принимать RHS как строку без дополнительного парсинга;
- для `value_kind: "string"` принимать RHS как строку как есть;
- для `value_kind: "typed_scalar"` разбирать RHS как отдельный YAML 1.2.2 value-node с тем же профилем типизации, что и `YAML frontmatter`;
- строковые значения, которые выглядят как `boolean` / `integer` / `null`, пользователь должен передавать в YAML-совместимой quoted-форме;
- зафиксировать это правило в пользовательской справке и в тестах.

Причина: это лучше всего согласуется со стандартом, который нормирует типизацию данных через YAML-профиль реализации.

### 3.4. Семантика `--content-file` и `--content-stdin`

Для `--set-file` baseline явно требует чтение содержимого "как есть". Для whole-body input это не проговорено отдельно.

Зафиксированное решение:

- `--content-file` и `--content-stdin` читать как строку без trim и без интерпретации escape-последовательностей;
- на этапе чтения не выполнять дополнительную нормализацию trailing newline или line endings;
- не добавлять неявную нормализацию содержимого, кроме той, которая уже обязательна для сериализатора итогового Markdown-документа.

### 3.5. Создание секций через `content.sections.<name>`

Write-namespace позволяет писать в `content.sections.<name>`, а значение path является телом секции без heading. Чтобы создать валидный Markdown, сериализатор должен сам создать heading.

Зафиксированное решение:

- канонически сериализовать секцию как heading формы `## <title> {#<label>}`;
- если в схеме у секции задан `title`:
  - брать первый допустимый `title` из канонического списка схемы;
- если `title` не задан:
  - использовать сам `label` как текст заголовка;
- тело секции вставлять без дополнительного заголовка, ровно как получено из CLI;
- порядок секций при создании и пересборке брать из `schema.entity.<type>.content.sections`.

Это решение не противоречит стандарту: валидатор обязан распознавать форму `{#label}`, а сравнение `title` выполняется только если `title` задан в схеме.

### 3.6. Совмещение whole-body и section write-path на `add`

Для `update` baseline прямо запрещает смешивать `--content-file` / `--content-stdin` / `--clear-content` с `content.sections.<name>`. Для `add` такой запрет не сформулирован.

Зафиксированное решение:

- запретить смешивание whole-body input и `content.sections.<name>` и для `add`;
- считать взаимоисключающими:
  - `--content-file` и любой `--set` / `--set-file` для `content.sections.<name>`;
  - `--content-stdin` и любой `--set` / `--set-file` для `content.sections.<name>`;
- конфликт отлавливать на этапе CLI/request validation до построения кандидата;
- возвращать `INVALID_ARGS` и не переходить к полной валидации сущности.

Такой подход делает `add` симметричным `update` и снимает неоднозначность результата при одновременном наличии whole-body и section-level patch.

### 3.7. Поведение `--dry-run`

Baseline показывает поле `dry_run`, но не фиксирует, как именно вести себя с `created`.

Зафиксированное решение:

- при `--dry-run` не писать на диск ничего;
- возвращать `dry_run: true`;
- возвращать `created: true`, если кандидат успешно прошел все проверки и был бы записан без `--dry-run`;
- возвращать тот же success-payload, что и у обычного `add`, то есть `created`, `entity` и `validation`;
- единственное семантическое отличие успешного `dry-run` от успешного обычного `add` — `dry_run: true` и отсутствие записи на диск.

### 3.8. Источник write-контракта для `add`

Зафиксированное решение:

- `add` не читает `schema.model` и не получает производный write-contract извне;
- единственный источник истины для write-contract в `add` — raw schema стандарта;
- допустимые write-path выводятся локально из raw schema:
  - `meta.<field>` для каждого поля из `meta.fields`;
  - `refs.<field>` только для тех полей из `meta.fields`, где `schema.type = entity_ref`;
  - `content.sections.<name>` для каждой секции из `content.sections`;
- built-in path, aggregate-path и expanded read-path отвергаются кодом по baseline-правилам:
  - `type`, `id`, `slug`, `created_date`, `updated_date`;
  - `content`, `content.raw`, `content.sections`;
  - `refs.*.id`, `refs.*.type`, `refs.*.slug`;
- `--set-file` не трактуется как общий "string input" механизм;
- `--set-file` разрешен только для `content.sections.<name>`;
- `--set-file` для `meta.<field>` и `refs.<field>` всегда отвергается с кодом `WRITE_CONTRACT_VIOLATION`, даже если итоговое значение логически является строкой;
- отдельная обязательность "пользователь обязан передать path X" не вводится;
- если после сборки кандидата не хватает обязательных данных сущности, это ловится только full validation и оформляется как validation-fail, а не как `INVALID_ARGS`.

### 3.9. Классификация request-level ошибок `add`

Зафиксированное решение:

- синтаксические и структурные ошибки запроса возвращаются как `INVALID_ARGS`;
- к `INVALID_ARGS` относятся:
  - отсутствие обязательных CLI-аргументов;
  - конфликтующие флаги;
  - дубли одного path;
  - некорректный синтаксис `path=value`;
  - нарушение `--require-absolute-paths`;
- нарушения write-контракта возвращаются кодом `WRITE_CONTRACT_VIOLATION`;
- к нарушениям write-контракта относятся:
  - path вне локально выведенного write-контракта;
  - запись в baseline-запрещенный path;
  - использование `--set-file` для path, где эта операция не разрешена;
- код `WRITE_CONTRACT_VIOLATION` используется для всех таких случаев единообразно.

### 3.10. Коллизия канонического пути

Зафиксированное решение:

- если после вычисления канонического path выясняется, что по этому path уже существует другой документ, команда должна завершаться кодом `PATH_CONFLICT`;
- `PATH_CONFLICT` не является ни `INVALID_ARGS`, ни `WRITE_CONTRACT_VIOLATION`, ни validation-fail;
- коллизия path считается конфликтом состояния workspace;
- при `PATH_CONFLICT` запись не производится, существующий файл не меняется, временные артефакты не остаются.

### 3.11. `exit_code` для domain-fail `add`

Зафиксированное решение:

- `WRITE_CONTRACT_VIOLATION` возвращается с `error.exit_code = 1`;
- `PATH_CONFLICT` возвращается с `error.exit_code = 1`;
- validation-fail итоговой сущности возвращается с `error.exit_code = 1`;
- различение этих случаев выполняется по `error.code`, а не по числовому `exit_code`.

### 3.12. Форма `refs` в успешном ответе `add`

Зафиксированное решение:

- в успешном `entity`-payload поле `refs.<field>` возвращается в сокращенной форме;
- минимальный и достаточный состав `refs.<field>` в ответе `add`:
  - `id`
- `type` и `slug` в `refs.<field>` для ответа `add` не возвращаются;
- response-контракт `add` для `refs` должен быть `id`-only.

### 3.13. Алгоритм вычисления `revision`

Зафиксированное решение:

- `revision` вычисляется по точным байтам итогового сериализованного Markdown-документа;
- в hash входят и `YAML frontmatter`, и body, и фактическая newline/formatting-политика сериализатора;
- одинаковый байтовый результат должен давать одинаковый `revision`;
- любое байтовое изменение итогового документа должно менять `revision`;
- `dry-run` должен вычислять тот же `revision`, который был бы у реальной записи.

### 3.14. Newline policy итогового Markdown

Зафиксированное решение:

- итоговый Markdown сериализуется с platform-native newline policy;
- на Unix-подобных системах используется `\\n`;
- на Windows используется `\\r\\n`;
- `revision` считается по байтам уже после применения platform-native newline policy;
- в пределах одной и той же платформы одинаковый логический документ должен давать одинаковый байтовый результат и одинаковый `revision`;
- cross-platform идентичность `revision` не гарантируется.

### 3.15. Body для типов без `content`

Зафиксированное решение:

- если у типа в raw schema отсутствует блок `content`, любой whole-body input запрещен;
- `--content-file` и `--content-stdin` для такого типа должны завершаться кодом `WRITE_CONTRACT_VIOLATION`;
- `content.sections.<name>` для такого типа также запрещены и завершаются кодом `WRITE_CONTRACT_VIOLATION`;
- такие типы рассматриваются как frontmatter-only с точки зрения `add`.

### 3.16. Поведение при параллельных `add`

Зафиксированное решение:

- baseline-реализация `add` не использует filesystem/workspace lock;
- команда работает по модели `snapshot -> build candidate -> final checks -> atomic write`;
- при гонке между параллельными `add` winner не нормируется;
- проигравший процесс должен завершаться одним из уже зафиксированных доменных конфликтов, если его кандидат больше не может быть записан корректно;
- отсутствие lock не должно нарушать требование наблюдаемой атомарности каждой отдельной записи.

### 3.17. Канонический порядок полей frontmatter

Зафиксированное решение:

- `YAML frontmatter` сериализуется в каноническом порядке полей;
- built-in поля всегда идут первыми в фиксированном порядке:
  - `type`
  - `id`
  - `slug`
  - `created_date`
  - `updated_date`
- затем идут поля, выведенные из `meta.fields`, в порядке их объявления в raw schema;
- сериализатор не должен полагаться на произвольный порядок map/dict-структур;
- одинаковая логическая сущность на одной платформе должна давать одинаковый порядок полей и одинаковый байтовый результат.

### 3.18. Сериализация whole-body против section-level input

Зафиксированное решение:

- `--content-file` и `--content-stdin` трактуются как источник уже готового body;
- если body задается whole-body input, `add` не должен перестраивать его по секциям, менять порядок секций или нормализовать headings;
- whole-body input только встраивается в итоговый документ вместе с каноническим frontmatter и затем проходит полную валидацию;
- если body задается через `content.sections.<name>`, итоговый body всегда собирается канонически по схеме;
- таким образом, у `add` есть два разных режима формирования body:
  - whole-body: сохранить пользовательскую структуру;
  - section-level: построить каноническую структуру.

### 3.19. Создание parent directories

Зафиксированное решение:

- если канонический path сущности указывает в несуществующие parent directories, `add` должен создать их автоматически;
- создание parent directories является частью той же операции записи;
- при неуспехе записи не должен оставаться partially-created результат, нарушающий наблюдаемую атомарность итогового состояния;
- `dry-run` не создает каталоги на диске.

### 3.20. Конфликт `slug`

Зафиксированное решение:

- конфликт `slug` с уже существующей сущностью того же типа не проверяется отдельным precheck-правилом;
- уникальность `slug` внутри типа проверяется только на этапе полной валидации кандидата;
- конфликт `slug` оформляется как validation-fail итоговой сущности, а не как `INVALID_ARGS` и не как отдельный domain-code.

## 4. Предлагаемая внутренняя архитектура

Реализацию лучше собрать из пяти слоев.

### 4.1. CLI-слой

Отвечает за:

- разбор аргументов;
- проверку взаимоисключающих опций;
- проверку `--require-absolute-paths` для filesystem path-аргументов;
- нормализацию повторяющихся `--set` / `--set-file`;
- формирование machine-readable ошибок уровня `INVALID_ARGS` для синтаксических и структурных ошибок.

### 4.2. Слой raw schema extraction

Отвечает за извлечение из схемы всего, что нужно именно `add`:

- описание типа сущности;
- `id_prefix`;
- `path_pattern`;
- `meta.fields`;
- `content.sections`;
- локально выведенный allowlist write-path;
- локально выведенный allowlist path для `--set-file`, совпадающий только с `content.sections.<name>`.

Внутренний результат этого шага должен строиться напрямую из raw schema и не зависеть от `schema.model`.

### 4.3. Слой workspace snapshot

Нужен для:

- поиска существующих сущностей;
- проверки уникальности `id`;
- проверки уникальности `slug` внутри типа;
- резолва `entity_ref`;
- вычисления `refs.*` для `path_pattern` и validation;
- выделения следующего `N` для `id`;
- ранней проверки коллизии итогового пути.

Минимальные индексы:

- `entities_by_id`;
- `slugs_by_type`;
- `max_suffix_by_type`;
- `resolved_refs_index` или эквивалентный механизм быстрого поиска целевых сущностей.

### 4.4. Слой candidate builder

Должен строить кандидата на запись в памяти:

- frontmatter;
- нормализованное представление body;
- вычисленный относительный POSIX-path;
- сериализованный Markdown;
- `revision`.

На этом слое write-namespace переводится в shape реальной сущности:

- `meta.<field>` -> top-level field frontmatter;
- `refs.<field>` -> top-level field frontmatter со строковым `id`;
- `content.sections.<name>` -> нормализованная Markdown-section.

### 4.5. Слой validator reuse

Критично не писать для `add` отдельную "облегченную" валидацию.

Нужно переиспользовать тот же движок, который покрывает требования стандарта и используется `validate`, чтобы один и тот же документ:

- одинаково проходил `add`;
- одинаково проходил последующий `validate`;
- отдавал diagnostics в одном shape.

## 5. Пошаговый план реализации

### Этап 1. Разобрать и провалидировать аргументы CLI

Сделать:

- зарегистрировать команду `add` и ее опции;
- сделать `--type` и `--slug` обязательными;
- сделать `--content-file` и `--content-stdin` взаимоисключающими;
- запретить смешивание `--content-file` / `--content-stdin` с `content.sections.<name>` в `--set` / `--set-file`;
- запретить `--set-file` для path, отличных от `content.sections.<name>`;
- собрать все `--set` и `--set-file` в единый список write-операций;
- отловить повтор одного и того же path между `--set` и `--set-file`;
- проверить filesystem path-аргументы относительно `--require-absolute-paths`;
- для `--format json` обеспечить нормативный envelope ошибок.

Проверки на этом этапе:

- отсутствие обязательных аргументов;
- неизвестные CLI-опции;
- пустой/некорректный `path=value`;
- конфликтующие опции;
- смешивание whole-body input с `content.sections.<name>`;
- относительный путь при `--require-absolute-paths`.

Ожидаемый результат:

- либо нормализованная структура `AddRequest`,
- либо немедленный `INVALID_ARGS` с exit code `2`.

### Этап 2. Загрузить raw schema и извлечь write-контракт типа

Сделать:

- загрузить schema file;
- извлечь описание `entity.<type>`;
- проверить, что `--type` существует;
- подготовить локальный маппинг допустимых write-path напрямую из raw schema.

На этом этапе нужно сразу вычислить:

- множество допустимых path для `add`;
- множество path, допустимых именно для `--set-file`;
- множество нормативно запрещенных path по baseline-правилам;
- контентную модель типа;
- правила `path_pattern`;
- правила `meta.fields`.

Если у типа отсутствует `content`, этот этап должен пометить whole-body input как недопустимый для `add`.

Если schema не может дать нормативный результат команды, возвращать top-level error уровня схемы с exit code `4`.

### Этап 3. Снять snapshot workspace и собрать индексы

Сделать:

- прочитать текущий workspace;
- определить существующие сущности по правилам стандарта;
- построить индекс `id`;
- построить индекс `slug` внутри каждого типа;
- вычислить `max_suffix_by_type`;
- построить индекс для резолва ссылок `entity_ref`.

Этот шаг нужен до построения кандидата, потому что без него невозможно:

- сгенерировать `id`;
- проверить ссылочную целостность;
- вычислить `refs.*` для `path_pattern`;
- заранее поймать дубликаты.

### Этап 4. Провалидировать и типизировать write-операции

Для каждой операции:

- проверить, что path принадлежит write-contract, локально выведенному из raw schema;
- проверить, что path не относится к built-in path, aggregate-path или expanded read-path, запрещенным baseline-правилами;
- проверить, что `--set-file` используется только для `content.sections.<name>`;
- проверить, что `--content-file` и `--content-stdin` разрешены для данного типа;
- разобрать значение согласно raw schema и типу path.

Если любой из этих пунктов нарушен, команда должна завершаться кодом `WRITE_CONTRACT_VIOLATION` с `error.exit_code = 1`, а не `INVALID_ARGS`.

Правила преобразования:

- `meta.<field>`:
  - для `typed_scalar` разбор RHS как YAML 1.2.2 value-node с тем же профилем типизации, что и `YAML frontmatter`;
- `refs.<field>`:
  - строковый `id` целевой сущности;
- `content.sections.<name>`:
  - строка тела секции без heading.

Важно: этот этап не заменяет full validation. Он только гарантирует, что patch-вход синтаксически допустим для `add`.

### Этап 5. Сгенерировать built-in поля кандидата

Сделать:

- взять `type` из `--type`;
- взять `slug` из `--slug`;
- провалидировать `slug` по regex стандарта;
- вычислить новый `id` по правилу, зафиксированному в разделе 3.1;
- выставить `created_date` и `updated_date` от одного значения `Clock`, интерпретированного как UTC day;
- убедиться, что сгенерированный `id` не конфликтует с уже существующим.

На этом же шаге удобно собрать базовый frontmatter-каркас будущей сущности.

### Этап 6. Переложить write-операции в модель сущности

Сделать:

- скопировать `meta.<field>` в top-level YAML fields;
- скопировать `refs.<field>` в top-level YAML field со строковым `id`;
- подготовить whole-body:
  - из `--content-file`;
  - или из `--content-stdin`;
  - или как пустую строку;
- если используется whole-body input, сохранить пользовательскую структуру body без перестройки по секциям;
- если используются `content.sections.<name>`, собирать body только из section-level input и канонического порядка схемы;
- привести секции к каноническому порядку схемы;
- сериализовать frontmatter в каноническом порядке, зафиксированном в разделе 3.17;
- сериализовать итоговое тело документа.

Если тип не содержит `content.sections`, но пользователь пытается писать в `content.sections.<name>`, это ошибка аргументов/контракта `add`.
Если тип не содержит `content`, пользователь не может использовать `--content-file` и `--content-stdin`.

### Этап 7. Вычислить канонический путь создаваемой сущности

Сделать:

- нормализовать `path_pattern` в форму `cases`;
- вычислить `when` слева направо;
- выбрать первый подходящий `use`;
- подставить built-in, `meta.*` и `refs.*`;
- получить относительный POSIX-path;
- проверить коллизию с уже существующим документом.

Критически важно вычислять путь по уже собранному кандидату, а не по частичным данным. Иначе можно получить рассогласование между сериализованным документом и местом записи.

Если итоговый path уже занят другим документом, команда должна завершаться кодом `PATH_CONFLICT` с `error.exit_code = 1` до начала записи.

### Этап 8. Запустить полную валидацию кандидата

Нужно валидировать не "операции", а уже готовую сущность целиком.

Проверить:

- built-in поля;
- `id_prefix`;
- уникальность `id`;
- уникальность `slug` внутри типа;
- `meta.fields` по `required`, `required_when` и `schema`;
- ссылочную целостность и `refTypes`;
- `path_pattern`;
- допустимость и вычислимость плейсхолдеров;
- `content.sections`, включая наличие обязательных секций и корректность `title`;
- closed-world frontmatter;
- общую согласованность документа со стандартом.

На выходе этапа нужно получить либо:

- валидного кандидата с `validation.ok = true`,
- либо набор diagnostics в том же минимальном shape, что и у `validate`.

Если validation не прошла:

- вернуть top-level `error`;
- использовать exit code `1`;
- по возможности вложить `error.details.validation.issues[]`;
- не создавать и не перемещать файлов вообще.

### Этап 9. Вычислить `revision` и подготовить response payload

После того как сериализованный Markdown окончательно стабилизирован:

- вычислить `revision` по фактическому состоянию документа;
- вычислить `revision` по точным байтам итогового сериализованного Markdown-документа;
- перед вычислением `revision` применить platform-native newline policy сериализатора;
- собрать `entity` в read-shape, который соответствует baseline:
  - built-in поля;
  - `meta`;
  - `refs` как object-view;
  - при необходимости прочие поля, которые входят в нормативный payload команды.

Для `refs.<field>` в response нужно возвращать object-view только с `id`.

### Этап 10. Реализовать `--dry-run`

Если `--dry-run` включен:

- пройти все этапы до конца, включая full validation и вычисление `revision`;
- не писать ничего на диск;
- вернуть тот же success-payload, что и обычный `add`, но c `dry_run: true`.

Проверки для `dry-run` должны быть такими же строгими, как для реальной записи. Иначе dry-run потеряет смысл как предиктор результата.

### Этап 11. Реализовать атомарную запись

Для реальной записи:

- создать недостающие parent directories целевого path;
- создать временный файл рядом с целевым путем или в том же filesystem;
- записать туда сериализованный Markdown;
- при необходимости выполнить `fsync`;
- атомарно переместить временный файл в целевой путь;
- при ошибке очистить временный артефакт;
- гарантировать отсутствие partially-written результата.

### Этап 12. Собрать финальный контракт ответа

Успешный JSON-ответ должен содержать:

- `result_state`;
- `dry_run`;
- `created`;
- `entity`;
- `validation`.

Для ошибок:

- top-level `error`;
- `error.code`;
- `error.message`;
- `error.exit_code`;
- при наличии `error.details.validation.issues[]`.

Текстовый вывод можно поддержать как convenience-режим, но все интеграционные тесты на контракт должны идти через `--format json`.

## 6. План интеграционных тестов

### 6.1. Общий подход

Интеграционные тесты должны быть черноящичными:

- каждый тест поднимает временный workspace;
- в workspace кладется минимальная schema и нужные fixture-документы;
- команда запускается как реальный CLI-процесс;
- проверяются:
  - JSON-ответ;
  - exit code;
  - состояние файловой системы после команды;
  - при необходимости последующий `validate` или прямое чтение созданного Markdown.

Все нормативные проверки нужно делать через `--format json`. Нестрогий text-вывод имеет смысл покрыть только smoke-тестом, если проект вообще поддерживает его для `add`.

### 6.2. Базовый fixture-набор

Для сценариев `add` достаточно подготовить одну опорную схему с двумя типами, например:

- `service`:
  - простой тип без обязательных ссылок;
- `feature`:
  - `id_prefix`;
  - `path_pattern`, использующий `slug`, а в одном сценарии и `refs.container.dir_path`;
  - `meta.fields.status` со строковым `enum`;
  - `meta.fields.container` с `schema.type: entity_ref` и `refTypes: [service]`;
  - `content.sections.summary`;
  - `content.sections.implementation`.

Такой fixture позволяет одним набором тестов покрыть:

- генерацию `id`;
- вывод допустимых write-path из raw schema;
- ограничение `--set-file` только секционными path;
- ссылочную целостность;
- `path_pattern`;
- работу с секциями;
- required/optional поля.

### 6.3. Обязательные успешные сценарии

#### T1. Минимальное создание валидной сущности

Setup:

- в workspace уже есть один `service`, на который можно сослаться.

Команда:

- `spec-cli add --format json --type feature --slug retry-window --set refs.container=SVC-1`

Проверить:

- exit code `0`;
- `result_state = "valid"`;
- `created = true`;
- `dry_run = false`;
- `entity.type = "feature"`;
- `entity.slug = "retry-window"`;
- сгенерирован `id` вида `FEAT-N`;
- `created_date` и `updated_date` заполнены и равны;
- дата соответствует значению тестового `Clock`, заданному в UTC;
- `entity.refs.container.id = "SVC-1"`;
- `type` и `slug` в `entity.refs.container` отсутствуют;
- на диске появился ровно один новый файл;
- созданный документ проходит последующий `validate`.

#### T2. Создание с типизированным `meta` через `--set`

Команда:

- `spec-cli add --format json --type feature --slug retry-window --set refs.container=SVC-1 --set meta.status=draft`

Проверить:

- `entity.meta.status = "draft"`;
- значение реально попало в frontmatter;
- порядок built-in полей во frontmatter канонический;
- итоговый документ валиден.

#### T2a. YAML-типизация `typed_scalar` в `--set`

Setup:

- fixture-схема содержит `meta`-поля типов `integer`, `boolean`, `null` и `string`.

Команды:

- `--set meta.priority=1`
- `--set meta.enabled=true`
- `--set meta.optional_note=null`
- `--set meta.literal_code='"001"'`
- `--set meta.literal_word='"true"'`

Проверить:

- `1` интерпретируется как integer;
- `true` интерпретируется как boolean;
- `null` интерпретируется как `null`;
- quoted-значения остаются строками;
- типизация совпадает с профилем `YAML frontmatter`;
- итоговый документ валиден.

#### T3. Создание whole-body через `--content-file`

Setup:

- файл тела с заранее известным Markdown.

Команда:

- `spec-cli add --format json --type feature --slug retry-window --set refs.container=SVC-1 --content-file /abs/path/body.md`

Проверить:

- тело записано без trim;
- структура whole-body сохранена без перестройки по секциям;
- `revision` соответствует hash точных байтов итогового сериализованного документа;
- newline policy итогового файла соответствует текущей платформе выполнения;
- валидация секций проходит или падает строго в зависимости от содержимого файла.

#### T4. Создание body через `--content-stdin`

Проверить то же, что и в `T3`, но для stdin.

#### T5. Создание секции через `content.sections.<name>`

Команда:

- `spec-cli add --format json --type feature --slug retry-window --set refs.container=SVC-1 --set content.sections.summary='short text'`

Проверить:

- секция реально создана в Markdown;
- heading создан сериализатором, а не ожидался от пользователя;
- секция стоит в каноническом месте относительно схемы;
- body собран канонически из section-level input;
- `validate` проходит.

#### T5a. Секция с `title` из схемы

Setup:

- у `content.sections.summary` в схеме задан `title`, например `["Summary", "Overview"]`.

Команда:

- `spec-cli add --format json --type feature --slug retry-window --set refs.container=SVC-1 --set content.sections.summary='short text'`

Проверить:

- сериализатор создает heading `## Summary {#summary}`;
- первый допустимый `title` берется из схемы детерминированно;
- пользователь не обязан передавать heading в CLI;
- итоговый документ проходит `validate`.

#### T6. Конфликт whole-body и section-path

Команда:

- whole-body задается через `--content-file`;
- поверх него задается `--set content.sections.summary=...`.

Проверить:

- команда завершается `INVALID_ARGS`;
- exit code `2`;
- запись не производится;
- full validation кандидата не запускается.

#### T7. `--set-file` для строкового path

Команда:

- `--set-file content.sections.summary=/abs/path/summary.md`

Проверить:

- содержимое файла прочитано как есть;
- секция создана без потери переводов строк;
- итоговая валидация проходит.

#### T8. `--dry-run`

Проверить:

- exit code `0`;
- `dry_run = true`;
- `created = true`;
- JSON payload содержит тот же `entity`, который был бы у реального `add`;
- JSON payload содержит тот же `validation`, что и реальный `add`;
- в файловой системе не появилось новых файлов;
- повторный реальный запуск без `--dry-run` создает тот же документ.

### 6.4. Обязательные негативные сценарии CLI и контракта `add`

#### T9. Отсутствует обязательный `--type`

Проверить `INVALID_ARGS`, exit code `2`, отсутствие записи.

#### T10. Отсутствует обязательный `--slug`

Проверить `INVALID_ARGS`, exit code `2`, отсутствие записи.

#### T11. Неизвестный `entity_type`

Проверить код `ENTITY_TYPE_UNKNOWN`, отсутствие записи.

#### T12. Одновременно заданы `--content-file` и `--content-stdin`

Проверить `INVALID_ARGS`, exit code `2`.

#### T13. Один и тот же write-path передан дважды

Варианты:

- два раза через `--set`;
- через `--set` и `--set-file`.

Проверить `INVALID_ARGS`, exit code `2`.

#### T14. Попытка писать в path вне локально выведенного write-контракта

Пример:

- `--set meta.unknown=value`

Проверить:

- команда завершается кодом `WRITE_CONTRACT_VIOLATION`;
- `error.exit_code = 1`;
- запись не производится.

#### T15. Попытка писать в нормативно запрещенный path

Примеры:

- `--set id=FEAT-999`;
- `--set content.raw=text`.

Проверить:

- команда завершается кодом `WRITE_CONTRACT_VIOLATION`;
- `error.exit_code = 1`;
- запись не производится.

#### T16. `--set-file` вне `content.sections.<name>`

Варианты:

- `--set-file meta.status=/abs/path/value.txt`
- `--set-file refs.container=/abs/path/value.txt`

Проверить:

- команда завершается кодом `WRITE_CONTRACT_VIOLATION`;
- `error.exit_code = 1`;
- запись не производится.

#### T16a. Whole-body input для типа без `content`

Setup:

- выбран тип сущности без блока `content` в raw schema.

Варианты:

- `--content-file /abs/path/body.md`
- `--content-stdin`

Проверить:

- команда завершается кодом `WRITE_CONTRACT_VIOLATION`;
- `error.exit_code = 1`;
- запись не производится.

#### T17. Нарушение `--require-absolute-paths`

Пример:

- `--content-file relative.md --require-absolute-paths`

Проверить `INVALID_ARGS`.

### 6.5. Обязательные негативные сценарии полной валидации

#### T18. Не переданы обязательные данные сущности

Пример:

- у `feature` обязателен `refs.container`, но команда не передает его.

Проверить:

- top-level `error`;
- exit code `1`;
- отсутствие записи;
- при наличии `issues[]` там есть диагностика по отсутствующему обязательному полю;
- ошибка возникает на этапе full validation кандидата, а не на precheck-этапе CLI.

#### T19. Некорректный `slug`

Пример:

- `Retry Window`

Проверить validation-fail и отсутствие записи.

#### T20. Дубликат `slug` внутри типа

Setup:

- в workspace уже есть `feature` с тем же `slug`.

Проверить validation-fail и отсутствие записи.

#### T21. Неразрешимая или типово некорректная `entity_ref`

Варианты:

- ссылка на отсутствующий `id`;
- ссылка на сущность не из `refTypes`.

Проверить validation-fail и отсутствие записи.

#### T22. Невалидная секция content

Пример:

- схема требует `summary`, но ни whole-body, ни section-path ее не создают.

Проверить validation-fail и отсутствие записи.

#### T23. `path_pattern` не может быть вычислен

Пример:

- выбранный кейс требует плейсхолдер от `refs.container.dir_path`, но ссылка не может быть резолвлена.

Проверить validation-fail и отсутствие записи.

#### T24. Коллизия итогового пути

Setup:

- в target path уже лежит другой документ.

Проверить:

- команда завершается кодом `PATH_CONFLICT`;
- `error.exit_code = 1`;
- существующий файл не меняется;
- новый файл не появляется.

### 6.6. Сценарии на детерминизм и атомарность

#### T25. Детерминированная генерация `id`

Setup:

- в workspace уже есть `FEAT-0`, `FEAT-1`, `FEAT-4`.

Проверить:

- новый `id` равен `FEAT-5`;
- дырка `FEAT-2` не переиспользуется;
- правило соответствует `max(existing N) + 1`.

#### T26. Атомарность при ошибке записи

Нужно искусственно спровоцировать file-write failure после успешной валидации.

Проверить:

- итоговый target file отсутствует;
- временный файл не остался;
- workspace после команды идентичен состоянию до команды.

#### T26a. Автоматическое создание parent directories

Setup:

- канонический path новой сущности указывает во вложенный каталог, которого еще нет в workspace.

Проверить:

- `add` успешно создает недостающие parent directories;
- итоговый файл появляется по каноническому path;
- документ проходит `validate`.

#### T27. `revision` меняется от фактического содержимого

Создать две сущности с разным итоговым сериализованным Markdown и убедиться, что `revision` различается.

#### T27a. `revision` стабилен в пределах платформы

На одной и той же платформе два одинаково сериализованных документа должны давать одинаковый байтовый результат и одинаковый `revision`.

#### T27b. Порядок frontmatter детерминирован

Проверить:

- built-in поля сериализуются в порядке `type`, `id`, `slug`, `created_date`, `updated_date`;
- поля из `meta.fields` идут после built-in;
- порядок `meta`-полей совпадает с порядком их объявления в raw schema;
- повторный запуск того же сценария на той же платформе дает тот же порядок полей.

## 7. Порядок выполнения работ

Рекомендуемая последовательность реализации:

1. Завести `AddRequest` и CLI parsing.
2. Реализовать извлечение write-контракта напрямую из raw schema.
3. Реализовать workspace snapshot и индексы.
4. Реализовать write-path parsing и type conversion.
5. Реализовать candidate builder и сериализацию content sections.
6. Реализовать full validation кандидата через общий validator.
7. Реализовать `dry-run`.
8. Реализовать атомарную запись.
9. Собрать JSON envelope успеха и ошибки.
10. Добавить интеграционные тесты блоками: happy path -> CLI errors -> validation failures -> atomicity.

## 8. Критерии готовности

Реализацию `add` можно считать завершенной, когда одновременно выполнены условия:

- все обязательные сценарии из раздела 6 проходят;
- успешный `add` создает документ, который проходит `validate`;
- любой validation-fail возвращает top-level `error` и не меняет файловую систему;
- `dry-run` дает тот же кандидат, что и реальный запуск, но без записи;
- `id`, даты, путь, body и `revision` вычисляются детерминированно;
- поведение по всем решениям раздела 3 зафиксировано в коде и тестах.
